Admin Manual

How operators review and approve Marketplace requests

This guide is for admins and operators who manage approval workflows, review pending requests, and ship the app through Netlify. It covers manual approvals, automatic approvals, plan changes, and the dashboard controls you can use from the live site.

Approval dashboard

The dashboard aggregates all approval tracks and shows anything waiting for action.

  • Open Approval Dashboard.
  • Use the Waiting for Approval section for manual entitlement and plan-change requests.
  • Each pending record has a direct approve button.
  • Use the focused entitlement and account fields to narrow the view.

Approval types

Three different approval tracks appear in the app.

  • Automatic offer approval requires customer-account approval before the start time.
  • Manual entitlement approval applies to ENTITLEMENT_CREATION_REQUESTED.
  • Plan change approval applies to ENTITLEMENT_PLAN_CHANGE_REQUESTED.

Manual approval steps

1. Find the record Open the dashboard or the entitlement status page and identify the entitlement ID or account ID that is waiting.
2. Confirm the state Make sure the record is actually pending and not already approved, rejected, or expired.
3. Approve it Use the dashboard button or call the backend endpoint for the relevant track.
4. Verify the result Refresh the dashboard or entitlement record and confirm the status changed to approved or accepted.

Useful endpoints

Entitlement approval POST /.netlify/functions/marketplace-entitlement-approval
Plan change approval POST /.netlify/functions/marketplace-plan-change-approval
Google Pub/Sub ingress POST /.netlify/functions/marketplace-pubsub
Dashboard data GET /.netlify/functions/approval-dashboard-data

Deployment and QA

Netlify deploy Production deploys use the checked-in netlify.toml configuration.
Regression suite Run npm run qa:full to execute the simulator, live regression checks, and browser smoke tests.
Rollback Keep the latest commit and tag history so you can return to the last known-good state quickly if needed.
If the dashboard queue looks wrong, verify the entitlement record in the backend store first, then re-run the QA suite before deploying again.
